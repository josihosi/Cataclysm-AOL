# Bandit smoke visibility mark seam v0 (2026-04-20)

Status: checkpointed done for now.

## Promotion audit verdict

The parked bandit concept chain was coherent enough to promote one narrow smoke-specific implementation slice, and that slice is now checkpointed.
What used to be fake was the signal seam: current bandit seams could reason about smoke-shaped leads, but they did not yet derive those leads from a bounded smoke adapter grounded in the repo's real fire/smoke/wind footing.

This contract was the smallest honest promotion that moved toward a future 500-turn proof without spawning another parked mystery-box feeder doc.

## Scope

1. Build a bounded smoke-specific adapter that turns existing fire/smoke/wind footing, or equivalent deterministic smoke input packets, into a coarse smoke exposure summary for bandit logic.
2. Turn that summary into smoke-family overmap mark writes or smoke-born lead generation on the current bandit mark-generation seam.
3. Keep smoke reads bounty-first and coarse: strong smoke should usually mean `worth scoping out`, not exact identity truth, free loot permission, or hard danger certainty.
4. Preserve the current product rule that sustained clear-weather smoke remains meaningfully long-range, several OMT away, with an outer cap around the existing concept starter range instead of collapsing into comically tiny OMT intuition.
5. Expose the smoke packet and resulting mark/lead path in reviewer-readable report output so the later 500-turn proof can reuse this seam instead of hand-authoring smoke clues again.

## Non-goals

- full player/basecamp visibility and concealment implementation
- light or searchlight signal adapters
- broad terrain-masking or multi-signal fusion law
- global offscreen smoke-field simulation outside the reality bubble
- the first full 500-turn proof itself

## Success shape

This item is good enough when:
1. a bounded smoke adapter exists from current smoke/fire/wind footing into coarse overmap-readable smoke signal state
2. deterministic coverage proves the key bounded distinctions: weak versus sustained smoke, clear versus degraded conditions, and long-range smoke staying plausible without turning every tiny fire into a world-map beacon
3. generated smoke marks or smoke-born leads feed the current bandit mark-generation / playback / evaluator seams reviewer-cleanly
4. the slice stays visibly bounded and does not smuggle in the broader visibility system, light/searchlight work, or the 500-turn proof packet

## Validation expectations

- current bounded validation passed via `make -j4 tests` and `./tests/cata_test "[bandit]"`
